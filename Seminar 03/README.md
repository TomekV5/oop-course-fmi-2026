# 03. Жизнен цикъл на обект. Външни ресурси в клас. Конструктори и деструктори. Композиция и агрегация

### Жизнен цикъл на обект

Всеки обект има жизнен цикъл:

1. **Създаване** - заделя се памет за обекта и неговите член-данни се инициализират.
2. **Използване** - обектът съществува в даден scope и върху него могат да се извикват член-функции.
3. **Унищожаване** - когато обектът излезе от scope-а си или бъде освободен с `delete`, той се разрушава.

Тази идея е важна, защото обикновено искаме обектът да е **валиден още от момента на създаването си** и да освободи правилно всички ресурси, които притежава, когато жизненият му цикъл приключи.

---

### Защо не е удобно да инициализираме обекти без конструктор

Нека разгледаме клас, при който първоначалното състояние се задава чрез setter-и, както правихме до момента:

```cpp
class Student {
private:
    char name[32];
    int age;

public:
    void setName(const char* newName) {
        if (newName == nullptr || strlen(newName) > 32) {
            return;
        }
        strcpy(name, newName);
    }

    void setAge(int newAge) {
        if (newAge < 0 || newAge > 120) {
            return;
        }
        age = newAge;
    }
    //...
};
```

Използване:

```cpp
Student st; // st is invalid currently
st.setName("Ivan");
st.setAge(20);
```

Този подход има няколко недостатъка:

- обектът съществува известно време в **неинициализирано** или **частично инициализирано** състояние
- лесно можем да забравим да извикаме някой setter
- не можем да гарантираме, че обектът е валиден веднага след създаването си
- при по-сложни класове често има зависимости между член-данните и не е добра идея те да се задават поотделно и на различни места

Следователно ни трябва механизъм, който да ни позволи да създадем обекта директно в коректно начално състояние.

---

## Класове с външни ресурси

Проблемът става още по-ясен, когато класът не съдържа само обикновени стойности, а управлява **външен ресурс**.

Под външен ресурс тук ще разбираме нещо, което не е просто част от фиксирания размер на самия обект, като например:

- динамично заделена памет
- отворен поток към файл

### Пример с динамична памет

```cpp
class IntArray {
private:
    int* data;
    std::size_t size;

public:
    // ...
    void setSize(std::size_t newSize) {
        // memory leak?
        size = newSize;
        data = new int[size];
    }
    // ...
};
```

Използване:

```cpp
int main() {
    IntArray arr; // arr.data is invalid currently
    arr.setSize(10);
    //...
    arr.setSize(20); // memory leak

    // we should delete[] the dynamically allocated array
}
```

На пръв поглед това изглежда подобно на предишния пример със setter-и, но тук вече имаме по-сериозни проблеми:

- преди `setSize` обектът няма валидно състояние;
- ако забравим да извикаме `setSize`, `data` остава неинициализиран указател;
- ако извикаме `setSize` повече от веднъж, ще загубим адреса на вече заделената памет и ще се получи **memory leak**
- никъде не е описано как и кога тази памет ще бъде освободена

Това показва защо при класове с външни ресурси не е достатъчно просто "после да зададем стойности". Нужно е:

- ресурсът да се придобие правилно още при създаването на обекта;
- ресурсът да се освободи автоматично при унищожаване на обекта.

Точно тук помагат конструкторите и деструкторите.

---

## Конструктор

**Конструктор** е специална член-функция, която се извиква автоматично при създаване на обект.

- извиква се **точно веднъж** - при създаването на обекта
- има **същото име като класа**
- **няма тип на връщане**
- служи за инициализация на член-данните и за установяване на валидно начално състояние на обекта

Пример:

```cpp
class Point {
private:
    int x;
    int y;

public:
    Point(int x, int y) {
        this->x = x;
        this->y = y;
    }
};
```

Използване:

```cpp
Point p(3, 4);
```

След създаването на `p`, обектът вече е в смислено начално състояние.

---

## Деструктор

**Деструктор** е специална член-функция, която се извиква автоматично при унищожаване на обекта.

### Основни свойства

- извиква се **точно веднъж** - когато обектът се "унищожава" (най-често в края на scope или при `delete`)
- има името на класа с `~` отпред
- няма параметри
- няма тип на връщане
- служи за освобождаване на ресурси и финални действия преди унищожаването на обекта

При класове с външни ресурси той е особено важен, защото чрез него освобождаваме ресурса, придобит/създаден от конструктора (например динамично заделената памет).

---

## Конструктор и деструктор при класове с външни ресурси

Нека пренапишем примера с динамичната памет по правилен начин:

```cpp
#include <cstddef>

class IntArray {
private:
    int* data;
    std::size_t size;

public:
    IntArray(std::size_t size) {
        this->size = size;
        data = new int[size];
    }

    ~IntArray() {
        delete[] data;
    }
};
```

Тук вече имаме ясен модел:

- при създаване на обекта, конструкторът заделя необходимата памет
- докато обектът съществува, той "притежава" и работи с тази памет
- при унищожаване на обекта, деструкторът освобождава паметта

Сега при създаване на обект от тип `IntArray`, динамичната памет се заделя правилно и съответно при края на жизнения цикъл на обекта, заделената динамичната памет се освобождава правилно, без memory leak.

---

## Кога се извиква деструктор

Деструкторът се извиква:

- когато се стигне края на scope-a, в който е създаден обекта
- когато извикаме `delete` за обект, създаден с `new`;
- когато извикаме `delete[]` за масив от обекти, създаден с `new[]`;
- при разрушаване на *временни* обекти

Пример:

```cpp
#include <iostream>

class Example {
public:
    Example() {
        std::cout << "Example()" << std::endl;
    }

    ~Example() {
        std::cout << "~Example()" << std::endl;
    }
};

int main() {
    {
        Example e;
        {
            Example e2;
        } // Destructor of e2

    } // Destructor of e
}
```

## Constructor overloading

Както обикновените функции, така и конструкторите могат да бъдат дефинирани с различни аргументи (overload-нати). Това означава, че един клас може да има няколко конструктора с различни параметри.

```cpp
class Rectangle {
private:
    int width;
    int height;

public:
    Rectangle() {
        width = 1;
        height = 1;
    }

    // square
    Rectangle(int side) {
        width = side;
        height = side;
    }

    // this is the same as:
    // Rectangle(int side) : Rectangle(side, side) {}

    Rectangle(int width, int height) {
        this->width = width;
        this->height = height;
    }
};
```

Използване:

```cpp
Rectangle r1;
Rectangle r2(5);
Rectangle r3(3, 7);
```

Това позволява на класа да предлага няколко смислени начина за създаване на обекти.

## Default constructor

**Default constructor** е конструктор, който може да бъде извикан **без аргументи**.

Пример:

```cpp
class Example {
public:
    Example() {
    }
};
```

Ако не сме дефинирали никакви конструктори, компилаторът автоматично генерира такъв по подразбиране.

### Кога ни трябва default constructor

Той е нужен, когато искаме да създадем обект без допълнителни параметри:

```cpp
Example obj;
```

Особено важен е при **масиви от обекти**.

### Пример с масив от обекти

```cpp
class A {
public:
    A() {
    }
};

int main() {
    A arr[5];
}
```

Тук компилаторът трябва да създаде 5 обекта от тип `A`. За всеки от тях трябва да има начин да бъде извикан конструктор без параметри. Тоест, компилаторът създава 5 обекта един до друг в паметта, използвайки техния default-ен конструктор.

### Какво става, ако няма default constructor

```cpp
class A {
public:
    A(int x) {
    }
};

int main() {
    A arr[5]; // compilation error
}
```

Този код няма да се компилира, защото компилаторът не знае как да създаде всеки елемент на масива без параметри.

### Защо това е така

При реда:

```cpp
A arr[5];
```

ние не подаваме аргументи за всеки елемент поотделно. Следователно компилаторът трябва да може да извика един и същ конструктор без параметри за всички елементи.

---

## Кога компилаторът генерира default constructor

Ако не сме дефинирали **никакъв** конструктор, компилаторът автоматично генерира default constructor.

Но ако ние дефинираме поне един друг конструктор, например:

```cpp
class A {
public:
    A(int x) {
    }
};
```

тогава компилаторът вече не генерира автоматично default constructor и изрази като:

```cpp
A a;
```

или:

```cpp
A arr[3];
```

няма да работят.

Тогава се налага ние сами да дефинираме default-ен конструктор.

## Конвертиращ конструктор

Конструктор с един параметър може да се използва за **неявно(имплицитно) конвертиране** от друг тип към нашия клас.

```cpp
class A {
public:
    A(int x) {
    }
};
```

```cpp
void f(A obj) {
}
```

Можем да извикаме:

```cpp
f(12); // this works, it creates an object of class A by calling the A(12) constructor
```

Тук компилаторът създава обект от тип `A` чрез конструктора `A(int)`.

### Защо това се нарича конвертиращ конструктор

Защото позволява стойност от един тип - тук `int` - да бъде преобразувана в обект от нашия тип.

### `explicit`

С ключовата дума `explicit` можем да забраним конструкторът да се използва за неявни преобразувания.

```cpp
class A {
public:
    explicit A(int x) {
    }
};

void f(const A& obj) {
}

int main() {
    // f(12); // compilation error
    A a(12);
    f(a);
}
```

Това често е добър стил, когато не искаме неочаквани автоматични преобразувания.

## Инициализация и присвояване

Важно е да различаваме две неща:

- **инициализация (initialization)** - задаване на начална стойност в момента на създаването;
- **присвояване (assignment)** - промяна на стойността на вече съществуващ обект.

Например в този конструктор:

```cpp
class Point {
private:
    int x;
    int y;

public:
    Point(int x, int y) {
        // this->x and this->y are already intialized
        this->x = x; // assignment
        this->y = y; // assignment
    }
};
```

член-данните `x` и `y` вече са били създадени, а в тялото на конструктора им присвояваме стойности.

В много случаи има по-естествен и по-правилен начин за директна инициализация на член-данните - чрез **member initializer list**.

---

## Member initializer list

Най-естественият начин да инициализираме член-данни в C++ е чрез **member initializer list**.

```cpp
class Point {
private:
    int x;
    int y;

public:
    Point(int x, int y) : x(x), y(y) /* initialization */ {
    }
};
```

Тук `x` и `y` се инициализират директно, вместо първо да бъдат създадени и след това да им се присвои стойност. В member initializer list могат да се извикват конструктори на член-данните, както и конструктори на самия клас.

### Защо е полезен initializer list

- по-ефективен е
- по-естествен е от гледна точка на жизнения цикъл на обекта
- в някои случаи е **задължителен**

### Кога initializer list е задължителен

Има член-данни, които не могат да бъдат зададени правилно в тялото на конструктора и трябва да се инициализират още при създаването им:

- `const` член-данни
- референции
- обекти от типове без default constructor

Пример с референция:

```cpp
class A {
private:
    int n;

public:
    A(int n) : n(n) {
    }
};

class B {
private:
    A& a;

public:
    B(A& a) : a(a) {
    }
};
```

Пример с `const` член-данна:

```cpp
class Example {
private:
    const int x;

public:
    Example(int x) : x(x) {
    }
};
```

## Ред на инициализация на член-данните

Член-данните се инициализират **в реда, в който са декларирани в класа**, а не в реда, в който са изписани в initializer list.

```cpp
class Example {
private:
    int a; // initialized first
    int b; // second

public:
    Example() : b(2), a(1) {
    }
};
```

Въпреки че в initializer list първо е написано `b(2)`, реално първо се инициализира `a`, после `b`, защото така са декларирани в класа.

Това правило съществува, за да има ясен и предвидим ред на създаване на обекта, а и може да има зависимост между някои от член-данните. Същото се отнася и за композирани обекти като член-данни.

---

## Композирани обекти и ред на извикване на конструктори

```cpp
#include <iostream>

class A {
public:
    A() {
        std::cout << "A()" << std::endl;
    }

    ~A() {
        std::cout << "~A()" << std::endl;
    }
};

class B {
public:
    B() {
        std::cout << "B()" << std::endl;
    }

    ~B() {
        std::cout << "~B()" << std::endl;
    }
};

class C {
public:
    C() {
        std::cout << "C()" << std::endl;
    }

    ~C() {
        std::cout << "~C()" << std::endl;
    }
};

class X {
private:
    A first;
    B second;
    C objectsArray[3];

public:
    X() {
        std::cout << "Constructor of X" << std::endl;
    }

    ~X() {
        std::cout << "Destructor of X" << std::endl;
    }
};

int main() {
    X obj; // A() B() C() C() C() X()
} // ~X() ~C() ~C() ~C() ~B() ~A()

```

Преди да започне тялото на конструктора на `X`, първо се създават неговите член-данни:

1. `first`
2. `second`
3. `objectsArray[0]`, `objectsArray[1]`, `objectsArray[2]`

И чак след това се изпълнява тялото на конструктора на `X`.

Следователно редът е:

- constructor на `A`;
- constructor на `B`;
- constructor на `C` - три пъти;
- constructor на `X`.

Това е така, защото обектът `X` не може да се счита за напълно създаден, преди всички негови член-данни да са създадени.

---

## Ред на извикване на деструкторите при композирани обекти

При разрушаване на обект от тип `X` първо се изпълнява тялото на деструктора на `X`, а след това се разрушават член-данните му в **обратен ред**:

1. `objectsArray[2]`, `objectsArray[1]`, `objectsArray[0]`
2. `second`
3. `first`

### Защо редът е обратен

Това е естественият и безопасен начин за "разглобяване" на обекта.

Ако при създаване първо сме построили едни части, а после други, то при разрушаване първо трябва да премахнем последно построените части. Това следва идеята:

- **last created, first destroyed**

Така, ако някой по-късно създаден обект зависи от по-рано създаден, зависимостите се разрушават в безопасен ред.

## Композиция и агрегация

### Композиция

При **композиция** един обект съдържа друг обект като своя част ("притежава" го) и обикновено контролира неговия жизнен цикъл (`owns-a`).

```cpp
class Engine {
};

class Car {
private:
    Engine engine;
};
```

Тук `engine` е част от `Car`. Когато се създаде `Car`, се създава и `engine`. Когато `Car` се разруши, `engine` също се разрушава.

### Агрегация

При **агрегация** един обект пази връзка към друг, но **не го притежава** и не управлява неговия жизнен цикъл, а само го "използва" (`uses-a`).

```cpp
class Engine {
};

class Car {
private:
    Engine& engine;

public:
    Car(Engine& engine) : engine(engine) {
    }
};
```

Тук `Car` използва вече съществуващ `Engine`, но не го създава и не го унищожава.

Същата идея може да се реализира и с указател:

```cpp
class Car {
private:
    Engine* engine;

public:
    Car(Engine* engine) : engine(engine) {
    }
};
```

### Основната разлика

- При **композиция** вътрешният обект е част от външния и обикновено живее точно колкото него. Външният обект контролира жизнения цикъл на вътрешния.
- При **агрегация** външният обект само сочи или реферира към друг обект, който може да живее независимо.
